1LLM Wiki란?

논문은 읽고 나면 흩어진다 — 몇 달 뒤엔 뭘 읽었는지도 가물가물하다. LLM Wiki는 읽은 논문 하나를 아래 세 가지 모습으로 남기고, 주제별로 이어 붙여 점점 커지는 한 덩어리로 쌓는 방식이다. 그래서 나중에 질문하면 웹이 아니라 내가 쌓은 이 덩어리가 답한다.

연구 아이디어를 대신 내주거나 버튼 한 번에 문헌조사를 끝내주는 도구가 아니다. 읽은 걸 잊지 않게 쌓는 방식이다.

한 논문이 남는 세 가지 모습

📄
papers/

원본 PDF

정확히 인용할 때.

📝
sources/

요약

빠르게 훑을 때.

🔗
wiki/

연결된 노트

다른 논문과 잇고 종합할 때.

세 곳이 같은 이름(park2026-overtaking)으로 묶여 한 편으로 추적된다. 분류는 미리 정하지 않고 논문 내용으로 부여하며, 쌓이면 세분화한다.

2문헌 수집 — 주제를 주면 논문을 모은다

연구 아이디어는 네가 들고 온다. 이 에이전트엔 주제로 lit review를 시킨다 — “X에 대해 리뷰해줘.” 그러면 ① 찾고(어떤 논문이 있나) ② 받고(그 PDF를) ③ 위키에 넣는다. ‘찾기’와 ‘받기’는 다른 일이다.

자동 · 무료 키 + KAIST망 사람 손 필요

① 찾기 (discovery) — 어떤 논문이 있나

arXiv자동·무료

search.py로 검색. 프리프린트(CS·물리·로보틱스 등)에 강하다.

OpenAlex자동·무료

search.py로 함께 검색. 거의 전 출판사 메타데이터 + 인용 그래프를 무료로 훑는다. 대부분의 ‘찾기’는 여기서 끝난다.

LeapSpace선택 · 사람 손

OpenAlex로 부족할 때만. ScienceDirect 본문 기반 의미검색. 에이전트가 ≤500자 영어 질문을 만들어 주면, 네가 사이트에서 돌려 답(DOI 포함)을 붙여넣는다.

sciencedirect.com/leapspace ↗

② 받기 (fetch) — 그 PDF를 다운로드

arXiv · OA키 불필요

fetch_paper.py — arXiv와 무료 공개본(OA)은 키 없이 바로. DOI는 OA를 먼저 시도한다.

Elsevier · Wiley · Springer키 + KAIST망

fetch_paper.py — OA가 없으면 출판사 API로. LeapSpace가 찾아준 Elsevier DOI도 여기서 받는다.

IEEE브라우저

fetch_ieee.py — 브라우저로 받는다(Playwright 선택 설치). 캠퍼스망/KAIST VPN이면 로그인 없이도 받힌다.

직접 넣기키 불필요

못 받는 논문은 PDF를 papers/에 복사만 해도 된다. 그다음 ingest는 똑같이 동작한다.

“Elsevier API가 있으면 LeapSpace는 필요 없나?” — 아니다. LeapSpace는 찾기, Elsevier API는 받기다. LeapSpace가 DOI를 찾아주면 그 DOI를 API로 받는, 앞뒤 단계라 오히려 같이 쓰면 시너지다. 다만 ‘찾기’는 OpenAlex도 무료·자동으로 하므로 LeapSpace는 선택이다.

3지식 사슬 — 받은 논문이 위키가 되는 길

받은 PDF에 “이 PDF ingest해줘” 한 마디면 아래가 한 번에 진행된다. 단계는 순서가 있고, 같은 이름(stem)으로 세 곳에 남는다.

1
원본 PDFpapers/
수집한 논문 PDF. 파일명 = <첫저자><연도>-<키워드>.pdf
↓
2
요약sources/<stem>.md
논문 1편 = 요약 마크다운 1편. 모르는 값은 unknown.
↓
3
구조화 노트wiki/<category>/
내용으로 주제 카테고리 부여. 개념·방법은 concepts/·methods/에 따로 적고 백링크.
↓
4
종합wiki/overviews/<category>.md
카테고리별 허브. 그 주제 노트들을 백링크로 모아 분야 지도를 만든다.
🎯쌓인 위키가 곧 내 연구의 토대가 된다 — 분석·발표·코드. 여기까지가 이 시스템이 노리는 결과다.
생성 순서 엄수: PDF → sources → wiki. 리뷰 한 편이 아니라 지식이 누적되는 한 덩어리를 만드는 게 목표다.

4시작하기

1
받기. 폴더 이름은 자유다(my-wiki 등). 이후 내 GitHub에 따로 올려도 된다. 
git clone https://github.com/hyunchul176/tupa-llm-wiki.git my-wiki
cd my-wiki
2
에이전트 열기. Claude Code 또는 Codex CLI를 이 폴더에서 연다. 세션 시작 시 AGENTS.md(룰)와 CLAUDE.md(진입점)를 자동으로 읽는다.
3
(선택) 키 넣기. 유료 출판사 본문을 받고 싶을 때만. 키가 없어도 arXiv·무료 공개본(OA)은 받힌다. 
cp secrets/api-keys.example.json secrets/api-keys.json   # 그다음 본인 키 입력
4
첫 주제로 lit review. 에이전트에게 “OOO 주제로 lit review 해줘” 하면 찾기→받기까지 알아서 한다. 받은 PDF는 “이 PDF ingest해줘”로 위키에 쌓는다. 
python scripts/search.py "vision language action humanoid"   # 찾기
python scripts/fetch_paper.py 2406.09246                      # 받기 (arXiv id / DOI)
python scripts/fetch_ieee.py fetch 10.1109/JSEN.2022.3156971  # IEEE (브라우저)

5룰 & 솔직한 한계

4대 룰 — 위키의 가치를 만드는 핵심

🚫

웹검색 금지(기본)

검색은 명시적으로 요청할 때만. 잡지식으로 빈칸을 채우지 않는다.

📚

위키 우선

답할 땐 먼저 sources/·wiki/를 근거로 한다.

🔍

모르면 PDF 재독

위키에 답이 흐릿하면 papers/의 원문을 다시 읽는다.

🙋

없으면 요청

근거 자료가 없으면 추측 말고 “이 PDF를 추가해 달라”고 한다.

솔직한 한계

알아둘 것
  • 자동 만능이 아니다. 한 방에 끝나지 않는다 — 매일 조금씩 쌓는 누적의 가치를 노린 도구다.
  • 출판사 본문은 환경에 달렸다. Elsevier·Wiley·IEEE 등은 기관 라이선스와 KAIST 망에 의존한다. 못 받는 건 PDF를 직접 넣으면 된다.
  • 요약은 틀릴 수 있다. LLM 요약·노트는 단서지 최종 근거가 아니다. 최종 근거는 언제나 papers/의 원문이다.
  • 분류는 자라며 바뀐다. 카테고리를 미리 못 정한다. 쌓이면서 세분화·재배치되는 게 정상이다.
