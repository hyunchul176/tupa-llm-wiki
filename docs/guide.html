TUPA LLM-Wiki · 개인 논문 지식베이스

웹이 아니라,
내가 쌓은 위키를
근거로 답한다.

읽은 논문을 PDF → 요약 → 구조화 노트 → 종합으로 쌓아, 흩어진 지식을 검색·연결 가능한 한 덩어리로 만드는 개인 지식 시스템. Claude Code 또는 Codex CLI를 이 폴더에서 열어 굴린다.

papers/PDF원본 논문
sources/요약논문 1편 = .md 1개
wiki/구조화 노트주제 분류·백링크
overviews/종합카테고리 허브
→내 연구분석·발표·코드
01

이게 뭔가

새로운 앱이 아니라, Claude Code/Codex 위에서 도는 운영 방식이다. 핵심은 한 가지 약속 — 질문에 답할 때 웹을 뒤지지 않고, 내가 직접 쌓아 검증한 위키를 근거로 삼는다. 그래서 매일 조금씩 쌓는 누적이 곧 가치다.

내 근거로 답한다

요약·노트·원문이 한곳에 쌓이면, 답은 잡지식이 아니라 내 자료에서 나온다. 근거가 없으면 “이 PDF를 넣어달라”고 요청한다.

주제로 분류

소속 연구실이 아니라 논문 내용으로 분류한다. 카테고리는 미리 정하지 않고, 쌓이면 세분화한다.

각자 → 그룹

먼저 최소 1개월 각자 위키를 굴려 흐름을 체득한다. 충분히 익으면 그룹 단위로 합쳐 공동 지식 지도로 키운다.

02

지식 사슬 — 논문 한 편이 지나는 길

아래 단계는 순서가 있다. 각 단계는 윗 단계 없이 존재할 수 없다. 같은 논문은 papers·sources·wiki 세 곳에서 같은 파일명(stem)을 써서 추적·중복방지한다.

STEP 1PDFpapers/
원본 논문 PDF. 직접 넣거나 수집 스크립트(03 참고)로 받는다. 파일명은 <첫저자><연도>-<키워드>.pdf.
STEP 2요약sources/<stem>.md
논문 1편 = 요약 마크다운 1편. 모르는 값은 얼버무리지 않고 unknown으로 둔다.
STEP 3구조화 노트wiki/<category>/
내용 기반으로 주제 카테고리를 정해 노트를 만든다. 등장한 개념·방법은 concepts/·methods/에 따로 적고 백링크로 잇는다.
STEP 4종합wiki/overviews/<category>.md
카테고리별 종합 허브. 그 주제의 노트들을 백링크로 모아 분야 지도를 만든다.
RESULT내 연구분석·발표·코드
쌓인 위키가 곧 내 연구의 토대가 된다. — 여기까지가 이 시스템이 노리는 결과다.
에이전트에게 PDF를 주고 “이 PDF ingest해줘” 한 마디면 STEP 1→4가 한 번에 진행된다. 생성 순서는 항상 PDF → sources → wiki.
03

수집 계층 — PDF를 구하는 길

사슬의 입구는 PDF다. 그 PDF를 구하는 4가지 길. 녹색은 키 없이 바로 되는 것, 흙갈색은 당신이 직접 해줘야 하는 곳(키·로그인). 받은 PDF는 모두 papers/에 stem 이름으로 저장된다 → 곧장 “ingest해줘”.

search.py 주제·키워드로 찾기. arXiv·OpenAlex 후보 목록을 보여준다(무료본 여부 포함). 받을 것을 골라 식별자를 아래로 넘긴다. 키 불필요
fetch_paper.py 식별자(DOI·arXiv id)로 받기. DOI는 무료 공개본(OA) 우선 → 없으면 출판사 API(Elsevier·Wiley·Springer). arXiv는 키 없이 바로. 출판사=키+KAIST망
fetch_ieee.py IEEE 등 키 없는 곳. 브라우저로 받는다. 캠퍼스망/KAIST VPN이면 로그인 없이도 받힌다. 브라우저·Playwright
직접 넣기 받기 어려운 논문은 PDF를 papers/에 복사만 해도 된다. 그다음 ingest는 똑같이 동작한다. 키 불필요
수집 스크립트는 표준 라이브러리만 쓴다(설치 불필요). fetch_ieee.py만 예외로 pip install playwright && python -m playwright install chromium 가 필요하다.
04

시작하기

클론 → 에이전트 열기 → (선택) 키 → 첫 논문. 여기까지면 위키가 돈다.

1

받기

폴더 이름은 자유다(my-wiki 등). 이후 내 GitHub에 따로 올려도 된다.

$ git clone https://github.com/hyunchul176/tupa-llm-wiki.git my-wiki $ cd my-wiki
2

에이전트 열기

Claude Code 또는 Codex CLI를 이 폴더에서 연다. 세션 시작 시 AGENTS.md(룰)와 CLAUDE.md(진입점)를 자동으로 읽는다.

3

(선택) 키 넣기

유료 출판사(Elsevier·Wiley·Springer) 본문을 받고 싶을 때만. 키가 없어도 arXiv·무료 공개본(OA)은 받힌다. 키 파일은 git에 안 올라간다.

$ cp secrets/api-keys.example.json secrets/api-keys.json # 그다음 본인 키 입력
4

첫 논문 넣기

주제로 찾거나 식별자로 바로 받은 뒤, 에이전트에게 “이 PDF ingest해줘” 한 마디.

$ python scripts/search.py "vision language action humanoid" # 후보 찾기 $ python scripts/fetch_paper.py 2406.09246 # arXiv id / DOI $ python scripts/fetch_ieee.py fetch 10.1109/JSEN.2022.3156971 # IEEE (브라우저)
05

4대 룰 & 솔직한 한계

위키의 가치를 만드는 핵심 룰, 그리고 이 시스템이 못 하는 것.

R1웹검색 금지(기본)검색은 명시적으로 요청할 때만. 잡지식으로 빈칸을 채우지 않는다.
R2위키 우선답할 땐 먼저 sources/·wiki/를 근거로 한다.
R3모르면 PDF 재독위키에 답이 흐릿하면 papers/의 원문을 다시 읽는다.
R4없으면 요청근거 자료가 없으면 추측 말고 “이 PDF를 추가해 달라”고 한다.
자동 만능이 아니다.

한 방에 끝나지 않는다. 매일 조금씩 ingest하는 누적의 가치를 노린 도구다.

출판사 본문은 환경에 달렸다.

Elsevier·Wiley·IEEE 등은 기관 라이선스와 KAIST 망에 의존한다. 못 받는 건 PDF를 직접 넣으면 된다.

요약은 틀릴 수 있다.

LLM 요약·노트는 단서지 최종 근거가 아니다. 최종 근거는 언제나 papers/의 원문이다.

분류는 자라며 바뀐다.

카테고리를 미리 못 정한다. 쌓이면서 세분화·재배치되는 게 정상이다.
